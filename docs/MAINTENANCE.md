# OMNIA-VALIDATION — Maintenance Guide

## Purpose

This document defines the maintenance rules for OMNIA-VALIDATION.

The goal is to keep the repository:

```text
runnable
testable
reproducible
schema-aware
historically honest
non-destructive
```

Core boundary:

```text
measurement != inference != decision
```

---

## 1. Maintenance Principle

OMNIA-VALIDATION preserves experimental evidence.

Maintenance must improve structure without rewriting history.

Correct maintenance:

```text
adds tests
adds documentation
adds schema validation
adds reproducibility checks
adds non-destructive wrappers
preserves original results
```

Incorrect maintenance:

```text
rewrites old results silently
deletes negative results
converts CHECK to PASS without revalidation
hides failures
removes boundary statements
treats formatting normalization as falsifiable validation evidence
```


Inspect changes:


## 4. Directory Roles

The repository uses different directories for different evidence layers.

```text
results/
```

Historical generated JSON result files.

These preserve old experimental outputs.

They may not follow the current canonical result envelope.

```text
results_enveloped/
```

Schema-normalized copies of legacy results.

These follow the canonical envelope:

```text
experiment
status
created_at_utc
boundary
payload
```

```text
data/
```

Bounded datasets and source-output records.

```text
examples/
```

Runnable validation scripts.

```text
docs/
```

Documentation, result reports, status documents, schema guides, and maintenance guides.

```text
omnia_validation/
```

Reusable Python package utilities.

```text
tests/
```

Automated tests.

---

## 5. Rule For `results/`

Do not rewrite historical result files casually.

The `results/` directory is a historical evidence layer.

Allowed actions:

```text
add new result files from new experiments
validate JSON parseability
document known limitations
preserve negative results
```

Avoid:

```text
rewriting old payloads
changing old statuses silently
deleting weak results
deleting failed results
converting old results to the new envelope in-place
```

If historical results need canonical schema compatibility, use:

```text
results_enveloped/
```

not direct modification of:

```text
results/
```

---

## 6. Rule For `results_enveloped/`

The `results_enveloped/` directory contains schema-valid wrappers around legacy result files.

Generated by:

```bash
python examples/wrap_legacy_results_in_envelope.py
```

Validate generated files:

```bash
pytest -q
```

or directly:

```bash
omnia-validation validate-result results_enveloped/<result_file>.json
```

Correct interpretation:

```text
results_enveloped/ provides schema-valid wrappers.
```

Incorrect interpretation:

```text
results_enveloped/ scientifically revalidates old experiments.
```

Wrapper files should preserve original content inside:

```text
payload.legacy_result
```

Legacy status, when present, should remain inside:

```text
payload.legacy_status
```

---

## 7. Regenerating `results_enveloped/`

Run:

```bash
python examples/wrap_legacy_results_in_envelope.py
```

Then run:

```bash
pytest -q
```

This checks:

```text
historical results are parseable JSON
enveloped results follow canonical envelope
wrapper generator still preserves legacy payloads
wrapper generator still uses CHECK
```

If `results/` gained new JSON files, `results_enveloped/` should be regenerated.

After regeneration, inspect:

```bash
git status --short
git diff --stat
```

Expected changes may include:

```text
new files in results_enveloped/
updated manifest
```

Do not accept unexpected deletion or mutation of historical results.

---

## 8. Wrapper Status Rule

Legacy wrappers use:

```text
status: CHECK
```

Reason:

```text
wrapping is format normalization
wrapping is not scientific revalidation
```

Do not change wrapper status to:

```text
PASS
```

just because a legacy file contains:

```text
status: PASS
```

The original status is preserved separately inside:

```text
payload.legacy_status
```

This rule prevents overclaiming.

## 9. Adding A New Validator

Follow:

```text
docs/VALIDATOR_AUTHORING_GUIDE.md
```

Recommended pattern:

```text
examples/<validator_name>_vN.py
data/<input_name>_vN.jsonl
results/<validator_name>_vN.json
docs/<VALIDATOR_NAME>_V<N>.md
docs/<VALIDATOR_NAME>_V<N>_RESULT.md
```

A validator should:

```text
use deterministic inputs when possible
write JSON results
preserve boundary language
avoid semantic overclaims
document PASS/CHECK/FAIL logic
```

Recommended result envelope:

```text
experiment
status
created_at_utc
boundary
payload
```

Use:

```python
from omnia_validation.metadata import result_envelope
from omnia_validation.schemas import require_valid_result_envelope
```

Before writing:

```python
require_valid_result_envelope(result)
```

## 10. Adding A New Result File

If the result is new and generated after the schema layer exists, prefer canonical envelope format.

Validate it:

```bash
omnia-validation validate-json results/<new_result>.json
omnia-validation validate-result results/<new_result>.json
```

If the result is intentionally legacy-format, document why.

Then regenerate enveloped results:

```bash
python examples/wrap_legacy_results_in_envelope.py
```

Run:

```bash
pytest -q
```

## 13. Updating Schema Rules

Schema source:

```text
omnia_validation/schemas.py
```

Schema documentation:

```text
docs/RESULT_SCHEMA.md
```

When changing schema behavior, update:

```text
tests/test_schemas.py
tests/test_enveloped_results.py
docs/RESULT_SCHEMA.md
docs/PACKAGE_API.md
docs/PROJECT_STATUS.md
```

Be careful: stricter schema rules may break legacy enveloped files.

If a stricter schema is added, decide whether it applies to:

```text
new canonical results only
results_enveloped/
specific payload families
all result files
```

Do not silently apply stricter rules to historical artifacts without documentation.

## 15. Negative Results Policy

Negative results must remain visible.

Do not delete results merely because they show:

```text
weak correlations
CHECK states
DRIFT regimes
CRITICAL local behavior
threshold instability
boundary sensitivity
metric collapse
failed hypotheses
```

These are part of OMNIA-VALIDATION.

A falsification-oriented repository becomes stronger when its failure boundaries are visible.

---

## 16. Status Policy

Canonical statuses:

```text
PASS
CHECK
FAIL
```

Use `PASS` when:

```text
the tested structural condition survived this validation step
```

Use `CHECK` when:

```text
partial instability exists
boundary condition exists
evidence is mixed
legacy normalization occurred
manual review is appropriate
```

Use `FAIL` when:

```text
artifact is invalid
source is missing
hash mismatch occurs
required validation condition fails
```

Do not use `PASS` to make a result look better.

Do not use `FAIL` merely because instability was detected.

Instability may be the measured phenomenon.

---

## 17. Legacy Status Policy

Legacy statuses are preserved.

They are not automatically converted.

See:

```text
docs/LEGACY_STATUS_MAPPING.md
```

Legacy status values remain inside:

```text
payload.legacy_status
```

Wrapper status remains:

```text
CHECK
```

unless a future validator-specific mapping is explicitly documented and tested.

## 18. Release Hygiene

Before any future release or public milestone:

```bash
pytest -q
ruff check omnia_validation tests
```

Also check:

```bash
git status --short
```

Recommended manual checks:

```bash
omnia-validation validate-result results_enveloped/legacy_result_envelope_manifest_v0.json
python examples/wrap_legacy_results_in_envelope.py
pytest -q
```

Do not publish a release if:

```text
tests fail
CI is red
result schema docs disagree with code
README links are stale
legacy normalization docs disagree with actual behavior
```

---

## 19. Token And Credential Safety

Never commit:

```text
GitHub tokens
API keys
private credentials
local machine paths containing secrets
```

Do not place tokens in:

```text
README.md
docs/
examples/
tests/
notebooks
```

If a token is used from Colab or another temporary environment, pass it through a hidden prompt and revoke it when no longer needed.

## 21. Non-Goal

Maintenance does not mean making the repository look perfect.

Maintenance means preserving evidence while improving reproducibility.

This repository should remain:

```text
experimental
bounded
falsification-oriented
failure-preserving
schema-aware
```

Final boundary:

```text
measurement != inference != decision
```
